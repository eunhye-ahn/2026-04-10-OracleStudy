# SQL 내장함수 정리

## 문자열 내장함수

- `length` : 문자 개수
- `substr(문자열, 시작위치, 자를개수)` : 문자 자르기
- `instr(문자열, 시작위치, 몇번째)` : 문자위치번호 검색
  - 시작위치 : `indexOf`, `lastIndexOf` 로도 가능
- `rpad(문자열, 출력개수, 대체문자)` : 문자 채우는 함수
- `replace` : 문자를 변경

---

## 숫자 내장함수

- `mod` : 나머지
- `ceil` : 올림 - 정수로 처리 → 총페이지
- `trunc` : 내림
- `round` : 반올림

---

## 날짜 내장함수

- `sysdate` : 오늘날짜
- `months_between` : 두 날짜의 개월 차
- `add_months` : 개월 등록 → 며칠인지

---

## 변환함수

- `to_char` : 숫자 or 날짜 → 문자열 변환
- `to_date(문자열, 형식지정)` : 문자열을 날짜로 변환 시, 형식을 지정하여 변환 가능

> 날짜 계산 시에는 date 객체로 변환 후 연산 → `+`, `-` 로 제어 가능

### 날짜 · 숫자 포맷

| 구분 | 포맷 |
|------|------|
| 숫자 | `9,999,999` |
| 년도 | `YY` / `YYYY` |
| 월 | `MM` |
| 일 | `DD` |
| 요일 | `DAY` |
| 시 | `HH` / `HH24` |
| 분 | `MI` |

---

## 기타 함수

- `nvl(칼럼, 값)` : null 값이 있는 경우 다른 값으로 변경
- `decode` : `=` 연산만 가능한 셀렉트문
- `case` : 가독성이 좋고 모든 연산이 가능한 셀렉트문
- `rank` / `dense_rank` : 순위

# 서브쿼리 (Subquery)

> 스칼라 서브쿼리 / 인라인 뷰는 **저장되지 않음** → 보안성 높음

| 종류 | 위치 | 저장 여부 | 재사용 |
|------|------|-----------|--------|
| 중첩 서브쿼리 | WHERE 절 | X | X |
| 스칼라 서브쿼리 | SELECT 절 | X | X |
| 인라인 뷰 | FROM 절 | X | X (일회성) |
| VIEW | — | ✅ DB 저장 | ✅ 여러 곳에서 재사용 |

---

## 1. 인라인 뷰 (Inline View)

> `FROM` 절 뒤에서 **테이블 대신** 사용하는 가상 테이블

### 특징
- 쿼리 실행 중 임시 결과 집합을 만들어 JOIN / 가공에 활용
- **일회성** → 저장되지 않음 → 보안에 유리

### 주요 사용 시나리오
1. `GROUP BY` / `WHERE` 사용이 불가능한 경우 해결
2. **집계 결과를 다시 조건으로 필터링**할 때
3. **TOP-N / 페이징** 처리
4. 복잡한 JOIN을 단순화하여 조인 수를 줄이는 경우

---

## 2. 스칼라 서브쿼리 (Scalar Subquery)

> `SELECT` 절 뒤에서 **가상 컬럼**으로 사용

### 특징
- 반드시 **결과값 1개, 컬럼 1개** 반환
- **각 ROW마다 실행** → 데이터가 많을수록 속도 저하

### 사용 기준

| 상황 | 권장 방식 |
|------|-----------|
| 소량 데이터 / 캐시 가능한 경우 | 스칼라 서브쿼리 (직관적) |
| 대량 데이터 | JOIN 또는 인라인 뷰 권장 |

---

## 3. VIEW

> **저장된 SELECT 쿼리** (가상 테이블)  
> 하나 이상의 테이블을 참조하며, SQL 문장을 DB에 저장

### 특징
- **재사용성** : 동일한 SQL을 여러 시스템에서 반복 사용할 때 적합
- **보안성** : 민감한 컬럼을 제외하고 공개 가능

### 실무에서 쓰는 이유
- 정규화된 DB는 테이블이 많이 분리됨 (보통 7개 이상)
  → 매번 복잡한 JOIN을 반복하지 않도록 **뷰로 묶어서 제공**
- 분석 툴에서 직접 테이블 대신 뷰를 사용하는 경향이 많음
- 프로젝트가 여러 개일 때 (게시판, 대댓글, 계층형 등)  
  → **뷰** 또는 **PL/SQL 프로시저**로 제작하여 공통 사용

### 생성 예시
```sql
CREATE VIEW vw_user_public AS
SELECT user_id, username, email  -- 민감 컬럼(password 등) 제외
FROM users;
```

## 요약 : 언제 무엇을 쓸까?

```
일회성, 임시 가공, 페이징         → 인라인 뷰
단순 조회, 소량 데이터 가상 컬럼  → 스칼라 서브쿼리
대량 데이터, 성능 중요            → JOIN / 인라인 뷰
여러 곳에서 재사용, 보안 필요     → VIEW
```

# 시퀀스 (Sequence)

> 테이블과 **독립적**으로 존재하는 번호 자동 생성 객체  
> 주로 **기본키(PK) 값 자동 생성**에 활용

---

## 생성 문법

```sql
CREATE SEQUENCE seq_name
    START WITH 1       -- 시작 번호
    INCREMENT BY 1     -- 증가 단위
    NOCACHE            -- 미리 번호를 생성하지 않고 처리
    NOCYCLE;           -- 최대값 도달 시 순환하지 않음
```

---

## 주요 옵션

| 옵션 | 설명 |
|------|------|
| `START WITH n` | 시작 번호 지정 (기본값 : 1) |
| `INCREMENT BY n` | 증가 단위 지정 (기본값 : 1) |
| `NOCACHE` | 번호를 미리 생성하지 않고 순서대로 처리 |
| `CACHE n` | n개의 번호를 미리 메모리에 생성 (성능 향상, 번호 공백 발생 가능) |
| `NOCYCLE` | 최대값 도달 시 더 이상 생성하지 않음 |
| `CYCLE` | 최대값 도달 시 다시 시작값으로 순환 |

---

## 사용 방법

```sql
-- 다음 값 가져오기 (호출할 때마다 증가)
seq_name.NEXTVAL

-- 현재 값 확인 (NEXTVAL 호출 후에만 사용 가능)
seq_name.CURRVAL
```

### 테이블 INSERT 예시

```sql
INSERT INTO store (store_id, store_name)
VALUES (seq_name.NEXTVAL, '맛있는 식당');
```

---

## 주의사항

- 시퀀스는 **테이블과 독립적** → 테이블 삭제해도 시퀀스는 남아있음
- 시퀀스 삭제 후 재생성하면 번호가 **초기화**됨
- `NOCACHE` 사용 시 번호 공백 없이 순서 보장, 단 성능은 `CACHE`보다 낮음
- 한번 증가한 `NEXTVAL`은 **롤백해도 되돌아오지 않음**

---

## 삭제 / 수정

```sql
-- 삭제
DROP SEQUENCE seq_name;

-- 수정 (START WITH는 수정 불가 → 삭제 후 재생성 필요)
ALTER SEQUENCE seq_name INCREMENT BY 2;
```
